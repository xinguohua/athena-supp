A. Algorithm 1 — Structural Mutation

Referenced from §4.3.1 Structural Mutation in the main paper.

Input: G⁺ = (V, E) (benign anchor graph), G⁻ (attack corpus), K (retrieval count), m (seed count), r (hop radius)

Output: mutated graph G̃ = (Ṽ, Ẽ)

1: {G⁻_k} for k=1..K ← Top-K over G⁻ in 𝒢⁻ by K_WL(G⁺, G⁻) 2: s(u,v) ← |T(u) ∩ T(v)| / |T(u) ∪ T(v)| 3: Cands ← ∅ 4: for k = 1 to K do 5: Seeds_k ← Top-m over (p, q) in V × V(G⁻_k) by s(p, q) 6: for all (p, q) in Seeds_k do 7: S, S′ ← {p}, {q}; Q ← [(p, q)] 8: φ ← {p ↦ q} 9: while Q ≠ ∅ do 10: (p_c, q_c) ← Q.dequeue() 11: for all q′ ∈ N_{G⁻_k}(q_c) \ S′ do 12: 𝒰 ← N_{G⁺}(p_c) \ S 13: if 𝒰 ≠ ∅ then 14: p′ ← argmax_{u ∈ 𝒰} s(u, q′) 15: S, S′ ← S ∪ {p′}, S′ ∪ {q′} 16: φ(p′) ← q′; Q.enqueue((p′, q′)) 17: end if 18: end for 19: end while 20: C ← {u ∈ V | min_{v ∈ S} d(u, v) ≤ r} \ S 21: E_{S′} ← {(u, v) ∈ E(G⁻_k) : u, v ∈ S′} 22: Cands ← Cands ∪ {(S, S′, E_{S′}, φ, C)} 23: end for 24: end for 25: (S, S′, E_{S′}, φ, C) ← LlmSelect(Cands) 26: Ṽ ← (V \ S) ∪ S′ 27: Ẽ ← {(φ(u), φ(v)) | (u, v) ∈ E, {u, v} ⊄ S} ∪ E_{S′} 28: return G̃ = (Ṽ, Ẽ)

B. Prompt Template — Edge Mutation

Referenced from §4.3.1 LLM-Guided Edge Mutation in the main paper.

Prompt: Edge Mutation
[Context]
An intermediate mutated graph G̃_mid has been obtained by replacing a benign subgraph S in anchor G⁺ with an aligned attack subgraph S′ from corpus G⁻. The internal attack edges of S′ must be preserved to maintain the attack causal chain. We ask the LLM to edit only the boundary edges to maximize stealth.
[Input]
Candidate boundary edges, partitioned into two classes:
    Class 1 — Cross-boundary edges (S′ ↔ C):
        Existing edges: [(src_type:attr, op, dst_type:attr), ...]
        Candidate new edges: [(src_type:attr, op, dst_type:attr), ...]
    Class 2 — Benign-internal edges (C ↔ C):
        Existing edges: [(src_type:attr, op, dst_type:attr), ...]
        Candidate new edges: [(src_type:attr, op, dst_type:attr), ...]
[Task]
For each candidate edge, decide one of:
(1) ADD — add this new edge to blend the attack into surrounding benign behavior, or to enrich benign context as natural noise;
(2) REMOVE — remove this existing edge if it exposes the attack or appears semantically out of place in the new context;
(3) KEEP — leave the edge unchanged.
Internal attack edges of S′ are not editable and must not appear in decisions.
[Output]
JSON: [{edge_id, action: ADD|REMOVE|KEEP, reason}].

C. Prompt Template — Semantic Mutation

Referenced from §4.3.2 Semantic Mutation in the main paper.

Prompt: Semantic Mutation
[Context]
The following attack process nodes are embedded in a provenance graph with their associated file/network nodes and r-hop benign context C.
[Input]
{n} process nodes:
    Node 1: attributes={attributes}, associated_nodes={file/network_attributes}, strategy={assigned_strategy}, preserve={critical_component}, C={context_triples}
    ...
    Node n: attributes={attributes}, associated_nodes={file/network_attributes}, strategy={assigned_strategy}, preserve={critical_component}, C={context_triples}
[Task]
Mutate each process node's attributes via its assigned strategy:
    A. Replacement: preserve the component indicated by preserve and replace the other with benign values guided by C.
    B. Rewriting: rewrite the entire command into a different expression guided by C that fits the surrounding benign context.
    C. Extension: preserve entire command, prepend/append benign operations from C.
For each mutated process node, also output updated attributes for any associated file or network nodes whose values change.

Constraints:
(1) generated values must be legitimate system commands, file names, or IP addresses;
(2) mutated values must be compatible with neighboring operations in C.
[Output]
JSON: [{node_id, new_attributes, associated_updates}].

D. Additional Hyperparameter Analysis (GNN Layers and Embedding Dimension)

Referenced from RQ6. Influence of Different Parameters in the main paper. The main paper reports sensitivity for time window length (T) and neighborhood size (r); analysis for GNN layers (L) and embedding dimension (D) is provided here.

GNN Layers (L)

This parameter refers to the number of layers in the GNN. As the model depth increases, detection performance exhibits a non-linear trend: shallow architectures (e.g., 2 layers) yield lower F1 scores due to limited representational capacity; a depth of 3 layers achieves the best trade-off between expressiveness and generalization, resulting in the highest F1; further increasing the depth to 4 layers or more leads to performance degradation. Memory consumption remains largely stable across different depth settings. Overall, a 3-layer model provides the most robust balance between detection performance and resource overhead.

Embedding Dimension (D)

Embedding dimension refers to the dimensionality of the embedding vectors, which controls the scale of node feature representations. Increasing the dimension from 16 to 64 significantly improves recall and F1, indicating enhanced structural representation. Further increasing the dimension to 128 and 256 degrades F1, as embeddings are expanded into weakly informative dimensions, reducing the separability between normal and anomalous samples under distance-based metrics. Memory consumption varies little with embedding dimension. Overall, a 64-dimensional embedding provides the best balance between detection performance and system overhead.

E.2 Local Deployment Recipe

For Qwen2.5-7B/14B and GLM-4-9B we use vLLM 0.6.x for batched inference on a single NVIDIA RTX 4090 (24 GB):

# Qwen2.5-14B example (16-bit weights, fits in 24 GB with 2k context) python -m vllm.entrypoints.openai.api_server \ --model Qwen/Qwen2.5-14B-Instruct \ --dtype bfloat16 \ --max-model-len 2048 \ --gpu-memory-utilization 0.92 \ --port 8000 # Qwen2.5-7B / GLM-4-9B: same command, swap --model # DeepSeek-V3 (671B) is too large for a single 4090; we used the public DeepSeek API for the reported numbers.

Average end-to-end latency reported in tab:llm-cost includes network round-trip for API models (GPT-4o, DeepSeek-V3) and local vLLM queuing for the others.

E.3 Verification Retry & Fallback Policy

When a generated mutation fails any of the four unified-verification checks (operation legality, attribute feasibility, imperceptibility, hardness), ATHENA does not silently drop the candidate. Instead:

  1. Regenerate (up to 3 retries) — the same prompt is re-sampled with a fresh seed; the verification failure mode (which check failed) is appended to the system message to bias the next sample away from the same error.
  2. Fallback to next candidate — if all 3 retries fail, the next-highest-ranked candidate from the LlmSelect output is used as the mutation target.
  3. Hard drop — if every candidate exhausts its retry budget, the snapshot is excluded from the augmentation batch for this epoch (this happens for < 2% of snapshots across all 5 LLMs in our runs).

The "Total Calls" column in tab:llm-cost reports the per-snapshot average call count (1 selection + 1 mutation + average retries), which directly reflects each model's verification pass-through rate.

E.4 USD Cost Methodology

The "USD / 1k Snap" column in tab:llm-cost is computed as follows:

  • GPT-4o — OpenAI public pricing as of June 2026 ($2.50 per 1M input tokens, $10.00 per 1M output tokens), multiplied by per-snapshot token usage × Total Calls × 1000.
  • DeepSeek-V3 — DeepSeek public API pricing as of June 2026, same formula.
  • Qwen2.5 (7B / 14B), GLM-4 (9B) — amortized RTX 4090 rental cost using the vast.ai reference rate (~$0.40/hour) × wall-clock time per snapshot × 1000.

Caveat. API prices change frequently; the absolute USD numbers are illustrative as of the paper's revision date. The per-call token counts and latencies in tab:llm-cost are physical quantities and remain reproducible regardless of pricing shifts.

H. Verification Ablation — Rating Protocol & Per-Variant Scores

Referenced from §V RQ2 (2) Attack Variant Reliability in the main paper. The main paper reports per-condition usability/F1/MMD p-values in Table V (tab:variant-usability); this section provides the rater recruitment, scoring rubric, per-condition Krippendorff's α, MMD test configuration, and a sample of per-variant scores.

H.1 Rater Recruitment

Three security analysts with 5+ years in threat hunting / SOC operations were recruited independently. None were involved in the design or implementation of ATHENA. Raters were compensated at a standard hourly consulting rate and signed an NDA covering the synthesized attack content.

H.2 Rating Rubric (1–5 Likert)

For each generated variant (rendered as a serialized provenance sub-graph with associated process commands, file paths, and network endpoints), raters scored realism on a 5-point Likert scale:

  • 5 — Highly realistic. Could plausibly appear in real audit logs from a benign or compromised production host. No suspicious artifacts.
  • 4 — Realistic with minor concerns. Plausible overall; minor oddities (uncommon flag combination, unusual but valid path) do not break believability.
  • 3 — Borderline. Some elements feel synthetic; would need additional context to judge.
  • 2 — Mostly unrealistic. Multiple signals suggest synthetic or hallucinated content (e.g., implausible command chain, malformed identifiers).
  • 1 — Infeasible. Cannot occur in a real system — impossible operations, non-executable commands, malformed paths/IPs, or violation of file-type semantics.

Usability is the share of variants scoring ≥4 (i.e., counted as production-grade realistic). Raters worked independently in a blinded protocol: variant order was randomized per rater, and condition/model identity was withheld.

H.3 Per-Condition Krippendorff's α

Overall inter-rater agreement is α = 0.83 (interval scale, treating Likert as ordinal-interval). Per-condition breakdown:

Condition Krippendorff's α --------- ---------------- Direct 0.85 w/o Op Legality 0.81 w/o Attr Feasibility 0.80 w/o Imperceptibility 0.84 w/o Hardness 0.83 Full 0.86 Overall (all 6 × 5) 0.83

α > 0.80 indicates substantial agreement across all conditions, supporting the validity of the usability scores reported in tab:variant-usability.

H.4 MMD Test Configuration

The Maximum Mean Discrepancy (MMD) test uses a Gaussian RBF kernel with bandwidth σ chosen via the median heuristic on the pooled embedding distances. Embeddings come from the trained 3-layer GIN encoder (64-dim) used in the main detection model. For each condition, we sample 500 synthesized attack snapshots and 500 real (benchmark) attack snapshots; p-values are computed from 1000 permutation rounds. p > 0.05 indicates we cannot reject the null hypothesis of identical distribution.

I. ATT&CK Mapping Failure Cases (STE & CTE)

Referenced from §V RQ3 (1) ATT&CK Technique Mapping in the main paper. The main paper reports Full-Enhanced's residual 9.4% Same-Tactic Errors (STE) and 5.7% Cross-Tactic Errors (CTE); representative cases for each error class are provided here, drawn from the failure traces of Full-Enhanced on the four benchmark datasets.

I.1 Same-Tactic Errors (STE)

STE occurs when the predicted technique shares the correct ATT&CK tactic but resolves to a different sub-technique. Root cause: highly overlapping action semantics within the same tactic confuse Sentence-BERT embeddings even after dual-side enhancement.

Case STE-1 — Persistence: T1547.001 vs T1546.015

  • Ground truth: T1547.001 Registry Run Keys / Startup Folder
  • Predicted top-1: T1546.015 Component Object Model Hijacking
  • Audit signature: process writes a registry key under HKCU\Software\Microsoft\Windows\CurrentVersion\Run then exits.
  • Why confused: both ATT&CK descriptions emphasize "register a startup/execution-trigger mechanism through the Windows registry," producing near-identical action triplets after technique-side distillation.

Case STE-2 — Execution: T1059.001 vs T1059.003

  • Ground truth: T1059.001 PowerShell
  • Predicted top-1: T1059.003 Windows Command Shell
  • Audit signature: cmd.exe spawns powershell.exe -nop -w hidden -c <base64>.
  • Why confused: the parent process is cmd.exe; both sub-techniques describe "interactive command interpreter execution," and the per-snapshot view does not distinguish parent-vs-child interpreter without sequence context.

Case STE-3 — Defense Evasion: T1070.004 vs T1070.001

  • Ground truth: T1070.004 File Deletion
  • Predicted top-1: T1070.001 Clear Windows Event Logs
  • Audit signature: process unlinks files under C:\Windows\System32\winevt\Logs\.
  • Why confused: the deletion target happens to be an event log directory; both sub-techniques aim at "removing forensic artifacts from disk," and the audit record only exposes the syscall (NtDeleteFile) without the higher-level intent.

I.2 Cross-Tactic Errors (CTE)

CTE occurs when the predicted ATT&CK tactic itself is wrong. Root cause: audit records expose only operation-level signatures (file open, network connect, syscall arguments) without payload semantics, so techniques across tactics that share the same operational shape become ambiguous.

Case CTE-1 — Exfiltration vs Command and Control: T1041 vs T1105

  • Ground truth: T1041 Exfiltration Over C2 Channel
  • Predicted top-1: T1105 Ingress Tool Transfer
  • Audit signature: SSH session, file /tmp/data.tar.gz read, socket send() to attacker IP.
  • Why confused: both manifest as a file ↔ network transfer over an encrypted tunnel. Without payload inspection, direction (outbound exfil vs inbound tool drop) is ambiguous from the audit-log view; both techniques' descriptions emphasize "use the existing C2 channel to move a file."

Case CTE-2 — Discovery vs Discovery (cross sub-tactic boundary): T1083 vs T1057

  • Ground truth: T1083 File and Directory Discovery
  • Predicted top-1: T1057 Process Discovery
  • Audit signature: short interleaved sequence ls /etc, ps -ef, cat /proc/cpuinfo, ls /home.
  • Why confused: the time-window snapshot boundary splits a longer reconnaissance pattern; the visible portion is dominated by ps//proc reads, biasing the top-1 toward process discovery.

Case CTE-3 — Credential Access vs Collection: T1003.008 vs T1005

  • Ground truth: T1003.008 OS Credential Dumping: /etc/passwd and /etc/shadow
  • Predicted top-1: T1005 Data from Local System
  • Audit signature: process opens /etc/passwd and /etc/shadow for read, copies content to /tmp/creds.txt.
  • Why confused: the audit record shows a generic file-read-then-write to local disk; without recognizing the security significance of the source paths, the technique resolves to the broader "data from local system" tactic.

Across all four datasets, STE and CTE residuals concentrate in tactics whose ATT&CK definitions share operational signatures (Persistence, Execution, Defense Evasion for STE; Exfiltration/C2, Discovery, Credential Access/Collection for CTE). Mitigating these residuals likely requires payload-level visibility or external context (process lineage, full command history) beyond the kernel-level audit stream.
