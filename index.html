A. Algorithm 1 ‚Äî Structural Mutation

The full pseudocode for structural mutation. Output GÃÉ is passed to semantic mutation (Section C).

Input: benign anchor graph G+; attack corpus ùí¢-; retrieval count N; hop radius r.
Output: structurally mutated graph GÃÉ.

Stage 1 ‚Äî Reference Graph Retrieval Retrieve the Top-N attack graphs {G-_1, ..., G-_N} from ùí¢- that are most similar to G+ under the Weisfeiler-Leman kernel. Stage 2 ‚Äî Aligned Region Search For each retrieved G-_n: pick the seed node pair (p‚ÇÄ, q‚ÇÄ) with the highest Jaccard similarity on node labels; grow an aligned subgraph pair (S, S') by repeatedly adding the best-matching unmatched neighbor pair, staying within r hops of the seed in G+; halt when no further matching pair can be added; record the average pair similarity of (S, S'). Keep the single (S, S') with the highest average pair similarity across all N candidates. Let œÜ be the node map S ‚Üí S'. Define C = r-hop neighborhood of S in G+ (the boundary context). Stage 3 ‚Äî Subgraph Replacement Remove S (and its internal edges) from G+; insert S' (and its internal edges) in place. For every edge of G+ with one endpoint p in S and the other endpoint outside S, remap p to œÜ(p) so the edge now reaches the corresponding node in S'. Call the result the intermediate graph. Stage 4 ‚Äî LLM-Guided Edge Mutation (prompt: Section B) Collect candidate edges between S' and C (both existing edges and plausible new edges). The LLM returns one of {ADD, REMOVE, KEEP} per candidate; apply the actions to the intermediate graph. Return the resulting graph as GÃÉ.

Notes. The LLM enters only in Stage 4. Stages 1‚Äì3 are deterministic: retrieval is a Top-N kernel ranking, alignment grows by Jaccard-best neighbor expansion, and the best (S, S') is chosen by argmax of average pair similarity.

B. Prompt Template ‚Äî Edge Mutation

For each candidate boundary edge between S' and C, the LLM decides one of {ADD, REMOVE, KEEP}. Internal edges of S' (the attack causal chain) and internal edges of C are not editable.

Prompt: Edge Mutation
[Context]
An intermediate graph has been obtained by replacing a benign subgraph S in anchor G+ with the aligned attack subgraph S'. Mutation is restricted to the boundary edges between S' and the surrounding context C; internal edges of S' and of C are not editable.
[Input]
Candidate boundary edges between S' and C, each described as a typed triple containing the source node type and attributes, the operation type, and the target node type and attributes:
    Existing edges to consider removing:
        [(src_type:attr, op, dst_type:attr), ...]
    Candidate new edges to consider adding:
        [(src_type:attr, op, dst_type:attr), ...]
[Task]
For each candidate edge, decide exactly one action:
(1) ADD ‚Äî add the proposed new edge, when it makes the cross-boundary connection plausible in the benign workflow of C;
(2) REMOVE ‚Äî remove the existing edge, when it would otherwise expose the attack or violate type/operation feasibility (e.g., a file-to-file write that would require an additional process-mediated injection path);
(3) KEEP ‚Äî leave the edge unchanged.
[Output]
JSON: [{edge_id, action: ADD|REMOVE|KEEP, reason}].

Example. Substitution inherits a write edge from the file node /usr/bin/sshd in S' to syslog in C. Action: REMOVE. Reason: a direct file-to-file write would require an additional process-mediated injection path (e.g., reflective loading) to be feasible.

C. Prompt Templates ‚Äî Semantic Mutation

Each process node is pre-classified as one of three cases ‚Äî Replacement (C.1), Rewriting (C.2), Extension (C.3) ‚Äî based on whether its command name and arguments are observed in the historical benign graphs. The classification is done externally, not by the LLM, because the full benign history is too large to fit in a prompt.

All three prompts share the same I/O: input is the process node's command name + arguments plus the boundary context C serialized as typed triples (src_type:attr, op, dst_type:attr); output is a rewritten command name + arguments. Propagation to associated file/network nodes is done mechanically post-hoc, not by the LLM.

C.1 Replacement Prompt

Triggered when exactly one of the command name or arguments is attack-specific (the other is benign-compatible). The attack-specific component must be preserved verbatim; the benign-compatible component is replaced with values that blend into the surrounding context C.

Prompt C.1: Replacement
[Context]
The attack process node below has exactly one component (either the command name or the arguments) marked as attack-specific and must be preserved verbatim; the other component is benign-compatible and must be replaced with values that blend into the surrounding context C.
[Input]
Process node:
    command_name = {name}
    arguments = {args}
    preserve = {"command_name" | "arguments"}  (the attack-specific component)
Context C (neighboring events as typed triples):
    [(src_type:attr, op, dst_type:attr), ...]
[Task]
Keep the component indicated by preserve exactly as given. Substitute the other component with benign values drawn from or compatible with C, so the resulting full command appears as a routine operation in the surrounding context.
[Constraints]
(1) generated values must be legitimate system commands, file names, or IP addresses;
(2) mutated values must be compatible with neighboring operations in C;
(3) the preserved component must appear verbatim in the output.
[Output]
JSON: {new_command_name, new_arguments}.

Example. Input: command_name="wget", arguments="http://mal.com/pay.sh", preserve="arguments" (the malicious URL), C = web-monitoring routines. Output: {new_command_name: "curl", new_arguments: "-H \"X-Health-Check: true\" http://mal.com/pay.sh"} ‚Äî the malicious URL appears as a routine monitoring target.

C.2 Rewriting Prompt

Triggered when both the command name and the arguments are individually benign-compatible (neither is attack-specific in isolation). The LLM rewrites the entire command into a different expression that fits the surrounding benign context C, while the attack causal chain remains intact through the surrounding structural mutation.

Prompt C.2: Rewriting
[Context]
For this attack process node, neither the command name nor the arguments is attack-specific in isolation. The full command may therefore be rewritten into a different expression that fits the surrounding benign context C; the attack causal chain remains intact via the structural mutation that placed this node.
[Input]
Process node:
    command_name = {name}
    arguments = {args}
Context C (neighboring events as typed triples):
    [(src_type:attr, op, dst_type:attr), ...]
[Task]
Rewrite the full command (new command name and new arguments) into a different expression that matches the dominant process type observed in C (e.g., if C is dominated by nginx processes, prefer a command consistent with a web-server workflow).
[Constraints]
(1) the rewritten command must be a legitimate executable + arguments combination;
(2) it must be plausible within the dominant process type in C;
(3) the rewritten command must be a distinct expression (not a trivial paraphrase) so it carries no residual signal from the original.
[Output]
JSON: {new_command_name, new_arguments}.

Example. Input: command_name="python", arguments="/tmp/script.py", C dominated by nginx processes. Output: {new_command_name: "php", new_arguments: "/var/www/cgi-bin/handler.php"} ‚Äî matches the surrounding nginx context.

C.3 Extension Prompt

Triggered when both the command name and the arguments are attack-specific and must be preserved verbatim. The LLM prepends or appends benign operations drawn from C to embed the malicious command within a normal workflow.

Prompt C.3: Extension
[Context]
For this attack process node, both the command name and the arguments are attack-specific and must be preserved verbatim. The mutation extends the command by prepending and/or appending benign operations drawn from C, embedding the malicious command within a normal workflow.
[Input]
Process node:
    command_name = {name}
    arguments = {args}
Context C (neighboring events as typed triples):
    [(src_type:attr, op, dst_type:attr), ...]
[Task]
Compose an extended command of the form <benign_prefix> && <original_command> && <benign_suffix>, where <original_command> is the input command_name followed by arguments, kept verbatim; and prefix/suffix are short benign commands drawn from or consistent with C (e.g., a status check or a routine log echo). Either prefix or suffix may be empty if a single side is sufficient to make the workflow plausible.
[Constraints]
(1) the original command name and arguments must appear exactly as given;
(2) the prefix/suffix commands must be legitimate and consistent with C;
(3) shell-level chaining must use plain operators (&&, ;) commonly observed in C.
[Output]
JSON: {new_command_name, new_arguments}  (where new_command_name may be a shell builtin such as sh -c and new_arguments contains the full extended sequence).

Example. Input: command_name="nc", arguments="-e /bin/sh attacker 4444", C dominated by nginx admin checks. Output: {new_command_name: "sh", new_arguments: "-c \"systemctl status nginx && nc -e /bin/sh attacker 4444 && echo \\\"connection test\\\"\""} ‚Äî the reverse shell appears as part of a network diagnostic routine.

D. Additional Hyperparameter Analysis (GNN Layers and Embedding Dimension)

Sensitivity analysis for GNN layers (L) and embedding dimension (D). T and r are analyzed in the main paper.

GNN Layers (L)

This parameter refers to the number of layers in the GNN. As the model depth increases, detection performance exhibits a non-linear trend: shallow architectures (e.g., 2 layers) yield lower F1 scores due to limited representational capacity; a depth of 3 layers achieves the best trade-off between expressiveness and generalization, resulting in the highest F1; further increasing the depth to 4 layers or more leads to performance degradation. Memory consumption remains largely stable across different depth settings. Overall, a 3-layer model provides the most robust balance between detection performance and resource overhead.

Embedding Dimension (D)

Embedding dimension refers to the dimensionality of the embedding vectors, which controls the scale of node feature representations. Increasing the dimension from 16 to 64 significantly improves recall and F1, indicating enhanced structural representation. Further increasing the dimension to 128 and 256 degrades F1, as embeddings are expanded into weakly informative dimensions, reducing the separability between normal and anomalous samples under distance-based metrics. Memory consumption varies little with embedding dimension. Overall, a 64-dimensional embedding provides the best balance between detection performance and system overhead.

Average end-to-end latency reported in tab:llm-cost includes network round-trip for API models (GPT-4o, DeepSeek-V3) and local vLLM queuing for the others.

E.3 Verification Retry & Fallback Policy

When a generated mutation fails any of the four unified-verification checks (operation legality, attribute feasibility, imperceptibility, hardness), ATHENA does not silently drop the candidate. Instead:

  1. Regenerate (up to 3 retries) ‚Äî the same prompt is re-sampled with a fresh seed; the verification failure mode (which check failed) is appended to the system message to bias the next sample away from the same error.
  2. Fallback to next candidate ‚Äî if all 3 retries fail, the next-highest-ranked candidate from the LlmSelect output is used as the mutation target.
  3. Hard drop ‚Äî if every candidate exhausts its retry budget, the snapshot is excluded from the augmentation batch for this epoch (this happens for < 2% of snapshots across all 5 LLMs in our runs).

The "Total Calls" column in tab:llm-cost reports the per-snapshot average call count (1 selection + 1 mutation + average retries), which directly reflects each model's verification pass-through rate.

E.4 USD Cost Methodology

The "USD / 1k Snap" column in tab:llm-cost is computed as follows:

  • GPT-4o (gpt-4o-2024-11-20) ‚Äî OpenAI public pricing for the pinned snapshot ($2.50 per 1M input tokens, $10.00 per 1M output tokens), multiplied by per-call token usage √ó Total Calls.
  • DeepSeek-V3 (DeepSeek-V3-0324) ‚Äî DeepSeek public API pricing for the pinned 0324 snapshot ($0.27 per 1M input tokens, $1.10 per 1M output tokens; non-cached pricing applied), same formula.
  • Qwen2.5 (7B / 14B), GLM-4 (9B) ‚Äî shown as -- in tab:llm-cost. These models run on locally-deployed weights (vLLM on a single RTX 4090) and incur no API expenditure, so no monetary cost is reported. Compute overhead for these models is captured by the per-snapshot latency columns.

Caveat. API prices change frequently; the absolute USD numbers are illustrative as of the paper's revision date. The per-call token counts and latencies in tab:llm-cost are physical quantities and remain reproducible regardless of pricing shifts.

G. Annotated Dataset & Source Code

The annotated dataset and the full source code are released together for reproducibility.

The annotated APT detection dataset and the full ATHENA source code are released together at:

G.1 Annotated Dataset

The repository's data/ directory contains the annotated APT detection dataset: manually verified malicious entity labels and ATT&CK technique labels for each malicious snapshot across DARPA E3, DARPA E5, ATLAS, and OpTC. The annotations were performed by three doctoral researchers in system security based on official attack reports (verification against ground-truth reports rather than subjective judgment; disagreements resolved by discussion). The release includes provenance graph snapshots, malicious entity labels, and ATT&CK technique mappings used in the experiments reported in ¬ßV of the main paper.

G.2 Source Code

The repository's src/ directory contains the full ATHENA implementation ‚Äî snapshot construction, LLM-guided graph augmentation (structural + semantic mutation), unified verification, adaptive contrastive learning (3-layer GIN + GRU + MLP), and global attack interpretation (Sentence-BERT mapping + LCS alignment). It also contains: (i) reproducible training/eval scripts for each dataset in tab:accrute; (ii) the prompt registry (Edge Mutation + Replacement / Rewriting / Extension) used in our LLM calls; (iii) the four unified-verification checks; (iv) hyperparameter configurations (time-window length T, neighborhood radius r, GNN layers L, embedding dimension D); and (v) the ATT&CK technique knowledge base with action-triplet enhancement.

H. Verification Ablation ‚Äî Rating Protocol & Per-Variant Scores

Rater recruitment, scoring rubric, per-condition Krippendorff's Œ±, MMD test configuration, and per-variant scores for the verification ablation reported in Table V (tab:variant-usability).

H.1 Rater Recruitment

Three security analysts with 5+ years in threat hunting / SOC operations were recruited independently. None were involved in the design or implementation of ATHENA. Raters were compensated at a standard hourly consulting rate and signed an NDA covering the synthesized attack content.

H.2 Rating Rubric (1‚Äì5 Likert)

For each generated variant (rendered as a serialized provenance sub-graph with associated process commands, file paths, and network endpoints), raters scored realism on a 5-point Likert scale:

  • 5 ‚Äî Highly realistic. Could plausibly appear in real audit logs from a benign or compromised production host. No suspicious artifacts.
  • 4 ‚Äî Realistic with minor concerns. Plausible overall; minor oddities (uncommon flag combination, unusual but valid path) do not break believability.
  • 3 ‚Äî Borderline. Some elements feel synthetic; would need additional context to judge.
  • 2 ‚Äî Mostly unrealistic. Multiple signals suggest synthetic or hallucinated content (e.g., implausible command chain, malformed identifiers).
  • 1 ‚Äî Infeasible. Cannot occur in a real system ‚Äî impossible operations, non-executable commands, malformed paths/IPs, or violation of file-type semantics.

Usability is the share of variants scoring ‚â•4 (i.e., counted as production-grade realistic). Raters worked independently in a blinded protocol: variant order was randomized per rater, and condition/model identity was withheld.

H.3 Per-Condition Krippendorff's Œ±

Overall inter-rater agreement is Œ± = 0.83 (interval scale, treating Likert as ordinal-interval). Per-condition breakdown:

Condition Krippendorff's Œ± --------- ---------------- Direct 0.85 w/o Op Legality 0.81 w/o Attr Feasibility 0.80 w/o Imperceptibility 0.84 w/o Hardness 0.83 Full 0.86 Overall (all 6 √ó 5) 0.83

Œ± > 0.80 indicates substantial agreement across all conditions, supporting the validity of the usability scores reported in tab:variant-usability.

H.4 MMD Test Configuration

The Maximum Mean Discrepancy (MMD) test uses a Gaussian RBF kernel with bandwidth œÉ chosen via the median heuristic on the pooled embedding distances. Embeddings come from the trained 3-layer GIN encoder (64-dim) used in the main detection model. For each condition, we sample 500 synthesized attack snapshots and 500 real (benchmark) attack snapshots; p-values are computed from 1000 permutation rounds. p > 0.05 indicates we cannot reject the null hypothesis of identical distribution.

I. ATT&CK Mapping Failure Cases (STE & CTE)

Representative failure cases behind Full-Enhanced's residual 9.4% Same-Tactic Errors (STE) and 5.7% Cross-Tactic Errors (CTE), drawn from the four benchmark datasets.

I.1 Same-Tactic Errors (STE)

STE occurs when the predicted technique shares the correct ATT&CK tactic but resolves to a different sub-technique. Root cause: highly overlapping action semantics within the same tactic confuse Sentence-BERT embeddings even after dual-side enhancement.

Case STE-1 ‚Äî Persistence: T1547.001 vs T1546.015

  • Ground truth: T1547.001 Registry Run Keys / Startup Folder
  • Predicted top-1: T1546.015 Component Object Model Hijacking
  • Audit signature: process writes a registry key under HKCU\Software\Microsoft\Windows\CurrentVersion\Run then exits.
  • Why confused: both ATT&CK descriptions emphasize "register a startup/execution-trigger mechanism through the Windows registry," producing near-identical action triplets after technique-side distillation.

Case STE-2 ‚Äî Execution: T1059.001 vs T1059.003

  • Ground truth: T1059.001 PowerShell
  • Predicted top-1: T1059.003 Windows Command Shell
  • Audit signature: cmd.exe spawns powershell.exe -nop -w hidden -c <base64>.
  • Why confused: the parent process is cmd.exe; both sub-techniques describe "interactive command interpreter execution," and the per-snapshot view does not distinguish parent-vs-child interpreter without sequence context.

Case STE-3 ‚Äî Defense Evasion: T1070.004 vs T1070.001

  • Ground truth: T1070.004 File Deletion
  • Predicted top-1: T1070.001 Clear Windows Event Logs
  • Audit signature: process unlinks files under C:\Windows\System32\winevt\Logs\.
  • Why confused: the deletion target happens to be an event log directory; both sub-techniques aim at "removing forensic artifacts from disk," and the audit record only exposes the syscall (NtDeleteFile) without the higher-level intent.

I.2 Cross-Tactic Errors (CTE)

CTE occurs when the predicted ATT&CK tactic itself is wrong. Root cause: audit records expose only operation-level signatures (file open, network connect, syscall arguments) without payload semantics, so techniques across tactics that share the same operational shape become ambiguous.

Case CTE-1 ‚Äî Exfiltration vs Command and Control: T1041 vs T1105

  • Ground truth: T1041 Exfiltration Over C2 Channel
  • Predicted top-1: T1105 Ingress Tool Transfer
  • Audit signature: SSH session, file /tmp/data.tar.gz read, socket send() to attacker IP.
  • Why confused: both manifest as a file ‚Üî network transfer over an encrypted tunnel. Without payload inspection, direction (outbound exfil vs inbound tool drop) is ambiguous from the audit-log view; both techniques' descriptions emphasize "use the existing C2 channel to move a file."

Case CTE-2 ‚Äî Discovery vs Discovery (cross sub-tactic boundary): T1083 vs T1057

  • Ground truth: T1083 File and Directory Discovery
  • Predicted top-1: T1057 Process Discovery
  • Audit signature: short interleaved sequence ls /etc, ps -ef, cat /proc/cpuinfo, ls /home.
  • Why confused: the time-window snapshot boundary splits a longer reconnaissance pattern; the visible portion is dominated by ps//proc reads, biasing the top-1 toward process discovery.

Case CTE-3 ‚Äî Credential Access vs Collection: T1003.008 vs T1005

  • Ground truth: T1003.008 OS Credential Dumping: /etc/passwd and /etc/shadow
  • Predicted top-1: T1005 Data from Local System
  • Audit signature: process opens /etc/passwd and /etc/shadow for read, copies content to /tmp/creds.txt.
  • Why confused: the audit record shows a generic file-read-then-write to local disk; without recognizing the security significance of the source paths, the technique resolves to the broader "data from local system" tactic.

Across all four datasets, STE and CTE residuals concentrate in tactics whose ATT&CK definitions share operational signatures (Persistence, Execution, Defense Evasion for STE; Exfiltration/C2, Discovery, Credential Access/Collection for CTE). Mitigating these residuals likely requires payload-level visibility or external context (process lineage, full command history) beyond the kernel-level audit stream.
